# ShelfFlow 面试讲解指南

## 1. 项目一句话介绍

ShelfFlow 是一个面向临期库存流转场景的双端系统，我把它按 production-ready 标准重构成了 Java 微服务项目，覆盖管理端商品/批次/履约，以及用户端目录/购物车/订单全链路。

## 2. 面试时建议先讲的主线

建议按照下面这条线讲，不要一开始就铺太多技术名词：

1. 业务背景
2. 为什么批次是核心模型
3. 为什么用户端和管理端要围绕订单打通
4. 后端服务怎么拆
5. 库存一致性怎么保证
6. 状态机怎么设计
7. 测试和验收怎么做

## 3. 业务背景讲法

可以这样讲：

> 这个项目的核心场景不是普通商城，而是临期库存流转。商品只是展示入口，真正影响价格、库存和履约的是批次。  
> 所以我把批次作为核心聚合，围绕它去设计用户下单、库存锁定、履约完成和状态流转。

这个讲法的好处是，面试官会更容易理解为什么你不是在做普通 CRUD。

## 4. 架构讲法

推荐这么讲：

> 我把系统拆成 gateway、auth-service、admin-service、user-service 四个服务起步。  
> gateway 统一暴露 `/api/**`。  
> auth-service 负责管理员认证。  
> admin-service 负责商品、批次、订单履约。  
> user-service 负责目录、购物车、订单。  
> 共享响应、错误码、枚举和鉴权上下文放在 service-common。

如果面试官追问 Spring Cloud Alibaba，可以补：

> 我接入了 Nacos Discovery 和 Config，默认本地模式仍然可以用固定地址启动；开启 `nacos` profile 后，Gateway 会切换到 `lb://服务名` 路由。  
> Sentinel 也做成可配置开关，网关按 routeId 限流，服务侧按核心接口资源限流，重点保护登录、验证码、下单、商品目录和 AI 问答这类高风险入口。

数据库迁移可以这样补：

> 我没有让 admin-service 和 user-service 各自启动时抢着建表，而是单独拆了 migration-service，用 Flyway 管理表结构版本。  
> 新环境先跑 migration-service，再启动业务服务；旧库通过 baseline-on-migrate 接入 Flyway 体系，后续所有结构变更都沉淀为版本脚本。

可以补一句：

> 我没有一开始就把 inventory、pricing、order、fulfillment 全部拆成独立服务，因为第一阶段更重要的是先把业务边界、状态流转和库存一致性做稳。

## 5. 为什么批次是核心

面试时重点讲这件事：

- 商品只描述“卖什么”
- 批次才描述“这批货什么时候过期、还剩多少、是否可售、价格如何变化”

可直接讲：

> 在这个系统里，价格、库存和履约的真正基础都不是 product，而是 inventory batch。  
> 所以我把批次状态、到期时间、可用库存、锁定库存、已售库存都放在批次模型里。

## 6. 库存一致性讲法

这是后端亮点，建议讲清楚三段：

### 6.1 下单

- 用户提交订单时，不直接扣减已售库存
- 高并发场景可先走 Redis Lua 批次预占，保护数据库事务提交前的短窗口
- 先锁定库存：`locked_quantity += quantity`
- 写入 `orders` 和 `order_detail`

### 6.2 取消

- 只允许特定状态取消
- 取消时释放锁定库存：`locked_quantity -= quantity`

### 6.3 完成

- 管理端履约完成时
- 把锁定库存结转为已售库存：
  - `locked_quantity -= quantity`
  - `sold_quantity += quantity`

推荐讲法：

> 我把库存变化和订单状态变化绑定在一起处理。  
> 下单是锁库存，取消是释放库存，完成履约才结转到已售库存。高并发下单时，我额外加了一层可配置 Redis Lua 原子预占，用短 TTL 记录正在提交的批次库存，数据库条件更新仍作为最终防线。
> 这样比下单就直接扣减已售更符合业务语义，也更方便处理中间状态。

### 6.4 事件驱动

> 订单提交、支付、取消都会先落库到 `order_event_log`，然后在事务提交后发布 RabbitMQ 领域事件。
> 这样下游通知、履约排队、经营分析可以异步订阅，不需要耦合进下单主链路。

### 6.5 支付幂等

> 支付接口不直接只改订单状态，而是先生成订单级支付单，使用 `userId + orderId` 作为幂等键。重复支付请求会复用既有支付结果，订单状态仍通过条件更新保证原子推进。

### 6.6 超时关单

> 我把未支付订单超时取消做成可配置后台任务，按批扫描待支付订单，事务内完成订单取消、库存释放和 system 审计事件写入，避免未支付订单长期占用临期库存。

另外可以补一句：

> 为了让链路可追踪，我给订单状态变化补了 `order_event_log`，把提交、支付、取消和管理端履约推进都记成结构化审计事件。  
> 这样线上如果出现库存争议或状态异常，可以直接按时间线回溯。

## 7. 状态机讲法

### 用户订单状态

建议直接背这条链：

```text
pending_payment -> to_prepare -> preparing -> ready_for_pickup -> completed
```

取消入口：

```text
pending_payment -> cancelled
to_prepare -> cancelled
```

说明点：

- 用户只能触发支付、取消
- 管理端负责履约推进
- 非法状态流转会返回 `conflict`

推荐讲法：

> 我没有把状态机写散在 controller 里，而是下沉到 policy / application service 里统一约束。  
> 这样后面增加支付确认、取消、完成等动作时，状态前置条件是集中可维护的。

## 8. 分层设计讲法

推荐直接按这一句：

> 我在 admin-service 和 user-service 里都按 `controller -> application service -> domain policy -> persistence mapper` 分层。

然后解释职责：

- controller：只处理 HTTP 协议和参数绑定
- application service：只做用例编排
- domain policy：做状态机、规则校验、排序白名单、库存约束
- persistence mapper：只负责 SQL 和数据映射

面试官如果追问“为什么这样拆”，可以回答：

> 这样做的核心价值是把业务规则和协议层、存储层分离。  
> 例如排序字段白名单、订单可支付状态、批次库存合法性这些规则，不应该散落在 controller 或 mapper 里。

## 9. 测试讲法

这是这个项目很加分的部分，建议主动讲：

### 当前测试层次

- 单元测试：policy / service 规则
- MVC integration test：鉴权、参数绑定、controller 行为
- H2 数据库级 integration test：真实 MyBatis / SQL / service 编排
- 前端构建：管理端和用户端 Next.js build

推荐讲法：

> 我没有只停留在 compile 通过。  
> 我把后端测试做到 H2 数据库级 integration test，并通过管理端、用户端前端构建保证基础交付质量。

## 10. 为什么保留 legacy backend

如果面试官问为什么还保留 `shelfflow-backend`：

推荐回答：

> 这个项目不是从零新建，而是从旧单体迁移。  
> 我保留 legacy backend 作为第一阶段运行依赖和迁移来源，同时把新主线收口到 Java 微服务。  
> 这样既能保证业务连续性，也能逐步把商品、批次、订单能力迁进新的服务边界。

这个回答体现的是工程迁移能力，而不是“不彻底”。

## 11. 你可以强调的亮点

面试时建议重点强调这些点：

- 不是普通 CRUD，而是围绕临期批次建模
- 有真实库存锁定 / 释放 / 结转
- 有双端业务闭环
- 有状态机
- 有订单审计事件流
- 有统一错误码和响应结构
- 有鉴权上下文
- 有 Nacos 注册发现 / 配置中心接入
- 有 Sentinel 网关和服务接口限流保护
- 有独立 Flyway migration service 管理数据库版本
- 有数据库级集成测试
- 有 Java integration test 和前端 build 验证

## 12. 面试问到不足时怎么答

不要说“没时间做”，而要说“下一步计划是什么”。

可以这样回答：

> 当前第一阶段已经完成管理端和用户端闭环，并且订单链路已经有审计日志。  
> 下一步我会继续补幂等控制、缓存策略、库存并发保护，以及把订单和履约进一步服务化。

这会比泛泛地说“后面还能优化”更有说服力。

## 13. 简历项目描述参考

可以直接改造成简历描述：

> 负责将旧单体业务重构为 Java 微服务项目，使用 Spring Boot / Spring Cloud Alibaba / MyBatis / Redis / MySQL，落地管理端与用户端双端闭环。  
> 设计以库存批次为核心的领域模型，完成商品、批次、购物车、订单、履约状态流转，以及库存锁定、取消释放、履约结转和订单审计事件追踪等关键链路。  
> 建立统一响应结构、错误码、鉴权上下文和数据库级集成测试，并通过前端构建与后端测试构建标准化验收体系。

## 14. 最后建议

真正面试时，不要试图一次把所有技术点讲完。  
最好的讲法是：

1. 先讲业务闭环
2. 再讲架构拆分
3. 最后讲库存一致性、状态机和测试

这样节奏更稳，也更容易把面试官带进你的设计逻辑里。
